# ARLS Finance Workflow Map

## Ownership Summary

- ARLS owns Finance 1차 live download generation.
- ARLS owns Finance final upload publish state and artifact storage.
- ARLS owns Finance 2차 download retrieval and per-user acknowledgement tracking.
- Finance state is implemented inside the main schedule router, not in a separate finance module.

## C. Finance Workflow

## 1. 1차 Download: Live Regeneration

### Backend functions

- `_collect_finance_review_export_context`
- `_build_monthly_export_workbook_from_contexts`
- `_create_finance_submission_batch`
- `_sync_finance_submission_state`
- `_build_finance_review_filename`

### Behavior

- `GET /api/v1/schedules/finance-submission/review-excel` generates a workbook from current live schedule/export context
- This is not artifact replay; it is regenerated from current truth at download time
- The route records a `review_download` batch
- State is updated to reflect that HQ review download has happened

## 2. Finance Upload Publish

### Preview path

- Route
  - `POST /api/v1/schedules/finance-submission/final-upload/preview`
- Functions
  - `_build_finance_submission_preview_payload`
  - `_create_finance_submission_batch`
- Behavior
  - Validates uploaded final workbook against current expected month/site/revision context
  - Persists preview/upload batch in `schedule_finance_submission_batches`
  - Stores uploaded artifact bytes in DB for later publish/download

### Apply/publish path

- Route
  - `POST /api/v1/schedules/finance-submission/final-upload/{finance_batch_id}/apply`
- Functions
  - `_build_finance_submission_preview_payload`
  - `_sync_finance_submission_state`
  - `_build_finance_submission_publish_version`
- Behavior
  - Revalidates the stored workbook bytes at apply time
  - Marks batch `applied` when valid
  - Advances the active published pointer on the state row

## 3. Publish Replace Semantics

### Current implementation

- Publish revision is generated by `_build_finance_submission_publish_version`
  - format: `publish:{batch_id}`
- State stores:
  - `active_final_batch_id`
  - `active_final_revision`
  - `active_final_source_revision`
  - `active_final_filename`
- New publish replaces the active pointer rather than merging rows

### Architectural meaning

- Finance final publish is pointer-based “latest active artifact” ownership
- 2차 download always resolves from the currently active final batch, not from an immutable per-user snapshot

## 4. 2차 Download: Latest Artifact Retrieval

### Backend functions

- `_load_finance_submission_final_artifact_for_site`
- `_build_finance_submission_final_bundle_workbook`
- `_record_finance_submission_download_ack`

### Behavior

- Single-site 2차 download returns stored final artifact for that site/month
- Multi-site or selected-scope 2차 download bundles the current published workbook from each selected site into one workbook
- Download acknowledgement is recorded after successful retrieval

### Frontend behavior

- `onScheduleFinanceFinalDownload`
  - uses `scope=site` for one site
  - uses `scope=selected` plus repeated `site_codes` for multi-site download

## 5. Recent Publish History

### Backend functions

- `_list_finance_submission_publish_history`
- `_build_finance_submission_status_payload`
- `_build_finance_submission_hq_workspace_payload`

### Behavior

- Status/workspace payloads expose recent applied final uploads
- History is trimmed to recent applied publish batches
- Current active batch is flagged separately from older applied publishes

## 6. Update-Needed State Logic

### Backend functions

- `_resolve_finance_submission_site_status`
- `_list_finance_submission_download_acks_by_site_code`

### State rules

- `파일 없음`
  - no current published final file
- `업데이트 필요`
  - a user has seen an older publish marker and a newer publish is now active
- `게시 완료`
  - current publish marker matches last acknowledged marker

## Architecture Review Notes

- Finance workflow is operationally mature enough to have dedicated tables, but the logic still lives inside the schedule-router monolith.
- Publish semantics are replace-pointer based, not version-branch based.
- The newest migration in the current worktree is finance-related (`020_schedule_finance_download_acks.sql`), so this area is an active change hotspot.
